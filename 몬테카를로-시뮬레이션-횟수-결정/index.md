## 한 줄 요약

10,000 → 50,000은 오차가 55% 줄지만, 50,000 → 100,000은 29%밖에 안 준다. 비용 대비 효용이 꺾이는 지점이 50,000이다.

---

## 배경 지식

### 몬테카를로 시뮬레이션이란

가능한 미래 시나리오를 N번 무작위로 뽑아서, 각 결과가 얼마나 자주 나오는지 세는 방법.  
이 코드에서는 "남은 LCK 경기를 ELO 기반 확률로 무작위 시뮬레이션 → 최종 순위 기록" 을 N번 반복한다.

```
실제 코드 흐름 (simulation.ts)
┌─────────────────────────────────┐
│ for i in 0..SIMULATIONS         │
│   ranked = simulateLckSeason()  │  ← 남은 경기 전부를 ELO로 승부 결정
│   for r, slug in ranked         │
│     rankCount[slug][r]++        │  ← 순위별 카운트 누적
│   (MSI, 월즈 로직 동일)          │
└─────────────────────────────────┘
확률 = count / SIMULATIONS
```

### 왜 많을수록 정확한가

각 팀의 확률 추정값은 이항분포(binomial)다.  
"N번 시뮬레이션 중 T1이 1위를 차지한 횟수"는 Binomial(N, p)를 따른다.

표준오차(Standard Error):

```
SE = sqrt(p × (1-p) / N)
```

N이 커질수록 SE가 줄고 → 추정값이 실제 확률에 가까워진다.  
단, **1/√N 비율**로 줄어서, N을 4배 늘려야 오차가 절반이 된다.

### 수렴 속도의 법칙 (√N 법칙)

오차 감소 = 1 - √(N₁ / N₂)

- 10,000 → 50,000: 1 - √(1/5) ≈ **55.3% 감소**
- 50,000 → 100,000: 1 - √(1/2) ≈ **29.3% 감소**

## 동작 원리 / 메커니즘

### 시뮬레이션 횟수별 95% 신뢰구간 폭 (±%)

확률 p별로 95% CI 반폭(= ±1.96 × SE)을 계산하면:

| 시뮬레이션 횟수 | p=50% (최대 분산) | p=30% (일반) | p=10% (낮은 확률) |
|---|---|---|---|
| 10,000 | ±0.98% | ±0.90% | ±0.59% |
| **50,000** | **±0.44%** | **±0.40%** | **±0.26%** |
| 100,000 | ±0.31% | ±0.28% | ±0.19% |

→ 50,000 기준, 어떤 확률 구간에서도 오차가 **±0.5% 이내**로 수렴한다.  
→ 100,000은 추가로 0.1%p 정도밖에 개선되지 않는다.

### 실행 복잡도 비교

LCK 10팀, 남은 경기 약 45경기라고 가정할 때:

```
시뮬레이션 1회당 연산:
  - 45경기 × 승부 결정 (random + ELO 계산) = 45회
  - 정렬: 10팀 = O(10 log 10) ≈ 33회
  - MSI 선발 + 월즈 로직 ≈ 수십 회

N=10,000 → 경기 결정 약 450,000회
N=50,000 → 경기 결정 약 2,250,000회
N=100,000 → 경기 결정 약 4,500,000회
```

JS 단일 스레드에서 50,000회는 수백 ms 수준, 100,000회는 1초 근접.  
API 응답으로 반환해야 하므로 1초를 넘기면 체감 지연이 발생한다.

## 해당 상황을 반복하지 않으려면 어떤 조치를 취해야 하나?

- **상수를 바꾸고 싶을 때**: `SE = sqrt(p*(1-p)/N)` 공식으로 먼저 계산해보고 결정. 오차 허용치를 먼저 정의한 뒤 N을 역산하는 게 정석.
- **더 빠르게 만들고 싶을 때**: N을 줄이지 말고 **분산 감소 기법**(Variance Reduction)을 검토. 대표적으로 층화 샘플링(Stratified Sampling)이 같은 N에서 오차를 줄여준다.
- **더 정확하게 만들고 싶을 때**: N을 늘리되, API 응답이 아니라 배경 작업(cron)으로 미리 계산해 DB에 저장하는 방식을 고려.

## 헷갈렸던 부분 / 함정

**"N을 2배 늘리면 오차도 2배 줄겠지"** → 틀렸다.  
오차는 **√N에 반비례**하므로, 오차를 절반으로 줄리면 N을 **4배** 늘려야 한다.

```
오차 ∝ 1/√N

N = 10,000 → 오차 ∝ 1/100
N = 40,000 → 오차 ∝ 1/200  (4배 늘렸더니 절반)
N = 100,000 → 오차 ∝ 1/316 (10배 늘렸더니 3.16배 감소)
```

→ 실제로 50,000 → 100,000 두 배를 늘렸을 때 오차 감소는 29%에 그친다. 직관과 많이 다름.

또 하나: **낮은 확률 구간(~10%)에서 오차가 오히려 작다.**  
SE = sqrt(p*(1-p)/N)이라서 p=0.5일 때 SE가 가장 크다.  
"특이한 일이 일어날 확률이 낮을수록 추정이 더 안정적"이라는 뜻.

## 응용·확장

- **다른 확률 이벤트에도 동일 공식 적용 가능**: 배당률 계산, 봇 예측 정확도 측정 등 이항 분포를 따르는 어떤 확률 추정에도 `SE = sqrt(p*(1-p)/N)` 하나면 N을 결정할 수 있다.
- **분산 감소 기법(Variance Reduction)**: 같은 N으로 더 정확하게 → 층화 샘플링, 중요도 샘플링(Importance Sampling). 현재 코드에서 적용하면 N을 줄이면서 같은 정확도 유지 가능.
- **사전 계산 + 캐싱**: 실행 비용이 문제라면 Vercel Cron으로 매 경기 후 미리 계산해 DB에 저장 → API는 조회만. (backlog 아이디어로 남겨둘 것)
